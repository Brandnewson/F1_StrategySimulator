\section{Designing the simulator}
Most F1 simulators discretise the race lap-by-lap, providing a base lap time and representing a faster car by subtracting from that time. A quasi-static approach using coordinate-space points and a vehicle model is also used, producing lap-time simulators of varying fidelity. Discretising lap-by-lap is computationally cheap but does not capture the stochastic nature of race outcomes under investigation. Modelling the physical properties of the car, as lap-time simulators do, is outside the scope of this project. This project discretises the race lap-by-lap while making overtaking a function of zone-specific difficulty values derived from real race telemetry, with a controlled stochasticity mechanism to test how robustly learned policies hold up as the environment becomes less predictable.

All configuration for the simulator is externalised to \texttt{config.json}. Reward semantics, observation features, stochasticity presets, and experiment budgets are all controlled without touching agent code. This keeps every algorithm comparison methodologically clean, since only the learning internals differ between runs.

\subsection{Modelling overtaking likelihood}
The track contains various overtaking zones at which agents decide whether to attempt an overtake or hold position. This is the fundamental mechanism through which race position changes in the simulator. The FastF1 library provides historical race telemetry for the 2023 Belgian Grand Prix, from which each overtaking zone's difficulty is gauged using real on-track data. Plotting the number of position changes against track distance reveals where overtakes cluster naturally and how contested each zone tends to be. Position changes are recorded at timing loop locations rather than the true on-track position of each car, so only the rough area of high overtaking probability can be inferred rather than an exact point. This is sufficient for assigning a relative difficulty value to each zone.

Nine overtaking zones are defined across the Spa-Francorchamps circuit, with difficulty values derived from the observed overtake density (see Figures~\ref{fig:overtakes-spa} and~\ref{fig:overtaking-mapping} in Appendix~\ref{sec:overtaking-figures}). La Source, the first braking zone, receives a low difficulty of 0.2 because overtakes are frequent there in practice. The remaining zones, including Raidillon, Les Combes, and Blanchimont, receive higher difficulty values of 0.7 to 0.9, as position changes rarely occur there in practice.

\subsection{Overtake success probability}
\label{sec:overtake-prob}

When an agent chooses to attempt an overtake, the simulator resolves the outcome by computing a success probability and sampling from it. The probability is computed as

\[
P_{\text{success}} = \text{clip}\!\left(\, p_{\text{base}} + \delta_{\text{risk}} + \delta_{\text{gap}} + \epsilon,\ p_{\min},\ p_{\max} \right)
\]

where each term is defined as follows.

\textbf{Base probability.} The zone difficulty $d \in [0,1]$ directly controls the base success rate. An easy zone with $d = 0.2$ yields $p_{\text{base}} = 0.8$, while a hard zone with $d = 0.9$ yields $p_{\text{base}} = 0.1$.
\[
p_{\text{base}} = (1 - d) \times s_{\text{base}}
\]
where $s_{\text{base}}$ is a scaling factor held at 1.0 throughout this study, so that stochasticity between levels is introduced solely through the noise term $\epsilon$ rather than by rescaling the base probability structure.

\textbf{Risk adjustment.} The agent selects from three risk levels, each adding a fixed offset to the probability.
\[
\delta_{\text{risk}} = \begin{cases}
-0.08 & \text{CONSERVATIVE} \\
\phantom{+}0.00 & \text{NORMAL} \\
+0.12 & \text{AGGRESSIVE}
\end{cases}
\]
A more aggressive attempt therefore shifts the probability upward but also incurs a larger failure penalty in the reward signal, creating a genuine risk-reward trade-off for the agent to learn.

\textbf{Gap modifier.} A closer following distance increases the probability of a successful overtake. The gap $g$ in kilometres is normalised against a reference distance of 0.1\,km, giving
\[
\delta_{\text{gap}} = \left(1 - \frac{g}{0.1} - 0.5\right) \times 0.3
\]

\textbf{Stochastic noise.} At higher stochasticity levels, Gaussian noise $\epsilon \sim \mathcal{N}(0, \sigma)$ is added to the probability before clipping. The noise standard deviation $\sigma$ is set by the active stochasticity level and is described in detail in Section~\ref{sec:stochasticity}. At s0, $\sigma = 0$ and outcomes are fully deterministic given the same state.

The final probability is clipped to $[0.02,\ 0.95]$ to avoid degenerate certainty in either direction.

A conservative attempt reduces the success probability by 0.08 relative to normal, while an aggressive attempt raises it by 0.12. However, failure penalties in the reward signal scale in the opposite direction. A failed conservative attempt costs $-0.5$, a normal failure $-1.0$, and an aggressive failure $-1.5$. The optimal risk level therefore depends on zone difficulty, current gap, and positional context.

\subsection{Finish-first reward contract}
The reward signal combines six components.
\[
R = R_{\text{outcome}} + R_{\text{persistent\_position}} + R_{\text{tactical}} + R_{\text{pace}} + R_{\text{tyre\_pit}} + R_{\text{penalty}}.
\]
Each component is normalised and weighted through \texttt{reward.weights}, \texttt{reward.normalization}, and \texttt{reward.component\_activation\_by\_complexity}.

The active components throughout this study are \texttt{outcome}, \texttt{persistent\_position}, \texttt{tactical}, and \texttt{penalty}. \texttt{Pace} and \texttt{tyre\_pit} are held at zero, keeping the learning problem focused on overtaking decision quality.

The objective hierarchy prioritises the finishing position. \texttt{outcome} is terminal and dominant, rewarding start-to-finish position gain at weight 2.0. \texttt{persistent\_position} provides dense shaping from position changes that survive decision resolution, at weight 0.1. \texttt{tactical} gives local feedback for overtake outcomes, with risk-scaled failure costs of $-0.5$, $-1.0$, and $-1.5$ for CONSERVATIVE, NORMAL, and AGGRESSIVE respectively, at weight 0.05. \texttt{penalty} remains available for invalid or catastrophic behaviour at weight 2.0. Overtaking is only worthwhile when it contributes to a better finishing position.

Each experiment in this project runs as a two-car race over five laps at Spa-Francorchamps. In single-agent phases, one DQN-family agent races against a fixed heuristic Base agent that does not learn. In multi-agent phases, a second concurrent DQN-family learner replaces or supplements the Base agent depending on the game structure under test.

\section{Stochasticity levels}
\label{sec:stochasticity}

The simulator exposes a controlled stochasticity pathway through three levels (s0, s1, and s2) that test how robustly learned policies hold up as the environment becomes less predictable. The active level is set via \texttt{stochasticity.active\_level}.

The stochasticity mechanism operates directly on the overtake success probability described in Section~\ref{sec:overtake-prob}. At each level, a noise standard deviation $\sigma$ is applied to the Gaussian noise term $\epsilon$. The remaining scaling factors (\texttt{base\_probability\_scale}, \texttt{risk\_modifier\_scale}, \texttt{gap\_modifier\_scale}) are held at 1.0 across all three levels so that only the outcome variance changes, not the underlying structure of the probability function.

Three levels are tested. s0 uses $\sigma = 0$, making outcomes fully deterministic given the same state, and serves as the primary ranking track. s1 uses $\sigma = 0.02$, introducing modest outcome variance while preserving the zone difficulty ordering. s2 uses $\sigma = 0.05$, a moderate perturbation sufficient to flip borderline outcomes and stress-test policy stability. The noise is additive, so the zone difficulty ordering is preserved in expectation even under s2, enabling measurement of robustness of learned ordering rather than just win rate under randomised outcomes. Using s0 as the ranking track follows established guidance that ranking decisions under high variance require larger sample budgets \citep{henderson2018deep}. Separating the ranking and robustness tracks keeps both interpretable at a controlled evaluation budget.

\label{sec:marl-methodology}

The investigation proceeds through phases~0 to~10, each gated by defined promotion criteria. Phases~0 and~1 are validation gates that confirm system correctness and a positive learning signal before the experimental matrix begins. Table~\ref{tab:phases} summarises the experimental matrix from phase~2 onwards. Phases~2 and~3 establish the single-agent and competitive baselines. Phases~4 to~10 introduce systematic variation in incentive structure, game structure, agent count, training schedule, and reward formulation. All configurations are controlled through \texttt{config.json} and require no changes to agent learning code.

\subsection{Phase 0: System integrity validation}
\label{sec:phase0-design}

Phase~0 validates the simulator, reward accounting, stochasticity implementation, and telemetry contract before any learning claims are made. Test Group~A checks simulator correctness, including position tracking, lap counting, and zone sequencing. Group~B validates stochastic probability distributions using repeated sampling to confirm the noise term produces the expected spread at each level. Group~C verifies the telemetry schema and reward component accounting.

\subsection{Phase 1: Single-agent smoke validation}
\label{sec:phase1-design}

Phase~1 runs vanilla DQN under a minimal budget of 25 training episodes and 20 evaluation races to confirm that a positive learning signal exists and that fairness checks pass before the full Phase~2 budget is committed. Three seeds are used at s0 and one seed each at s1 and s2. The gate criteria require win rates above 0.5 across all s0 seeds, zero fairness violations, and an interpretable degradation direction across stochasticity levels. These criteria are intentionally conservative, as Phase~1 is a signal check rather than a convergence measurement.

\subsection{Phase 2: single-agent DQN-family baseline}

Phase 2 establishes the DQN-family reliability baseline under a controlled single-agent contract. Each variant (vanilla, double, dueling, rainbow-lite) is trained and evaluated independently against the fixed Base Agent, with identical reward schema, observation contract, stochasticity level, episode budgets, and seeds across all variants. The purpose is to measure each variant's raw strategy quality and stochasticity robustness before multi-agent interaction is introduced, following the recommendation to isolate algorithm-level effects before attributing differences to interaction dynamics \citep{henderson2018deep}. Phase 2 produces the per-algorithm stochasticity profiles that inform algorithm selection for all subsequent phases.

The shared training configuration was determined prior to Phase 2 using Autoresearch \citep{karpathy2024autoresearch}, an AI-driven tool that iteratively proposes and evaluates single hyperparameter changes over 30 automated iterations, each scored across 3 seeds with 50 training runs. This ensures the episode budget, learning rate, network width, and exploration schedule are empirically grounded rather than an arbitrary choice that could favour one variant.

\subsection{Phase 3: independent learner MARL}
\label{sec:phase3-methodology}

Phase 3 introduces genuine MARL by replacing the fixed Base Agent with a second concurrent DQN-family learner. Both agents train simultaneously, each maintaining its own independent replay buffer and updating its own network parameters each episode. This is the independent learner (IL) formulation, the standard baseline in the MARL literature for operationalising non-stationarity in value-based multi-agent settings \citep{foerster2017stabilising,hernandezleal2019survey}.

IL is chosen as the first MARL baseline because it preserves genuine non-stationarity. It also provides a clean anchor from which future centralised-training with decentralised-execution methods such as QMIX can be measured \citep{rashid2018qmix}, and its integration required only a runtime profile change rather than any redesign of the simulator engine. Phase~3 is gated on Phase~2 passing all promotion-gate criteria.

\subsection{Phase 4: Alpha reward mixing}
\label{sec:alpha-mixing}

The alpha reward mixing mechanism modifies the reward received by each agent as follows.

\[
R_{\text{mixed}} = (1 - \alpha) \cdot R_{\text{own}} + \alpha \cdot R_{\text{teammate}}
\]

At alpha of zero each agent receives only its own reward. At alpha of one each agent receives only its teammate's reward. Intermediate values produce partial alignment between the two agents' incentives. Five alpha values are tested (0.0, 0.25, 0.50, 0.75, and 1.0) to identify whether there is a threshold beyond which cooperative incentive causes learning to collapse rather than cooperate. This sweep distinguishes a gradual performance degradation at high alpha from a structural passivity failure, which is the core empirical question of RQ2.

\subsection{Phase 5: Non-zero-sum game design}
\label{sec:nonzerosum}

Phases~2 to~4 use a two-agent zero-sum race. The sum of position changes across both agents is structurally fixed, so one agent's gain always comes at the other's expense.

Phase~5 changes the game structure by adding a fixed heuristic Base adversary as a third competitor. Both DQN agents can then improve simultaneously by outperforming the adversary, breaking the zero-sum constraint and allowing genuine cooperative advantage to be measured. An ablation within Phase~5 replaces each DQN agent with a Base agent in turn, isolating whether game structure or policy quality drives any observed improvement.

\subsection{Phases 6 and 7: Multi-team scaling}
\label{sec:multiteam}

Phase~6 extends the non-zero-sum setting to four DQN agents grouped as two teams of two, plus a single Base adversary. Phase~7 tests intermediate configurations of three and five agents to identify the count at which cooperative incentive transitions from beneficial to harmful, and whether any alpha value can recover cooperative performance at that scale. These two phases together test whether the alpha mixing findings from Phase~4 and Phase~5 generalise beyond the two-agent case, or whether cooperative incentive breaks down as the credit assignment problem grows with team size.

\subsection{Phase 8: Curriculum alpha scheduling}
\label{sec:curriculum}

Phase~8 applies a three-stage training schedule rather than a fixed alpha value throughout. Agents train for 100 episodes at alpha of zero to establish individual policies, then alpha increases linearly over 300 episodes to the target value, followed by a final 100 episodes at the target. The motivation is to test whether the passivity failure observed at fixed alpha of 0.75 results from cooperative reward overwriting individual policies before they are established, rather than from a structural incompatibility between cooperative incentives and independent learning.

\subsection{Phase 9: LLM semantic baseline}
\label{sec:llm-baseline}

Phase~9 introduces a zero-shot large language model as a non-learning comparator. The agent receives the race state as a natural language description including zone difficulty, gap to the car ahead, current position, and laps remaining, and outputs a structured action specifying whether to attempt an overtake and at which risk level. No training, replay buffer, or gradient update is used. The purpose is to establish whether domain reasoning alone can match the performance of a trained DQN agent. The result serves as an interpretability anchor. If a zero-shot LLM matches a trained agent, the learning problem may not have required adaptive strategy at all, which would undermine the basis for the MARL investigation.

\subsection{Phase 10: Difference rewards}
\label{sec:difference-rewards}

Phase~10 tests difference rewards as an alternative cooperative formulation. Alpha mixing is a heuristic that aligns incentives without formally separating individual contributions, and the passivity failure it can produce is fundamentally a credit assignment problem. Difference rewards address this directly by giving each agent credit only for its counterfactual contribution to collective performance \citep{wolpert2002optimal}.

\[
D_i = G(\mathbf{z}) - G(\mathbf{z}_{-i})
\]

where $G(\mathbf{z})$ is global performance under the full joint action vector and $G(\mathbf{z}_{-i})$ is global performance when agent~$i$'s contribution is removed. Because computing the true counterfactual requires simulating the race without agent~$i$, which is computationally costly, a mean-field approximation is used instead.

\[
R_{\text{diff},i} = \Delta_i + (\Delta_i - \bar{\Delta}_{\text{team}})
\]

where $\Delta_i$ is agent~$i$'s position delta and $\bar{\Delta}_{\text{team}}$ is the mean delta across all team members. Whether this approximation preserves the cooperative gradient property of the original formulation is the central question of Phase~10.

\section{Project management}

Development proceeds in research sprints aligned to the experimental phases, version-controlled with Git. Each sprint covers one phase. Design precedes implementation, the gate criteria act as the acceptance condition, and a branch is merged to the main line only once the gate passes. The main branch therefore always reflects a gate-passed simulator state.

Each run embeds a configuration snapshot in its output metrics, so any experiment is reproducible from the repository state at the time it ran. Commit messages document phase-level findings at meaningful checkpoints, including zone behaviour observations and collapse events.

The original project plan prioritised a minimal viable product before any MARL complexity was introduced. Phases~0 to~2 fulfil that objective directly. A working single-agent simulator with validated metrics was in place before a second learning agent was added. That order mattered when Phase~3 produced its first unexpected collapses. The diagnostic infrastructure was already trusted, so attributing collapse to replay buffer contamination rather than a simulator defect was possible without ambiguity. Having the gate requirement formalised removed the option to defer this step under time pressure.

Development was iterative throughout. Each phase was designed to produce a usable result independently, so the project had real stopping points at every gate rather than a single binary pass or fail at submission. It also meant that bugs introduced in one phase could be attributed to that phase's changes rather than to a compounding stack of untested code. The three defects caught in Phase~0 illustrate this. All three would have been substantially harder to isolate had they surfaced during a Phase~3 collapse.

The plan adapted when the central research question changed. The original aim was to study agent responses to stochasticity. That became a precursor question once the stochasticity baseline was established. The phase-gated structure made this pivot possible without losing continuity. The infrastructure remained intact and the interpretation shifted. The buffer period at the semester transition provided the space to reframe the research question without pressure on the experimental timeline.

The original project deliverables map directly to the phase structure. Deliverable~3 (minimal viable product with basic metrics) corresponds to Phases~0 and~1. Deliverable~4 (expanded state space, varied parameters, and algorithm comparison) corresponds to Phases~2 through~10. Deliverable~5 (final report with comparative analysis) is the present document. The gate structure meant each deliverable was verifiable at its checkpoint rather than assessed only at final submission.
