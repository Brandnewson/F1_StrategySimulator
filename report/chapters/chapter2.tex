\chapter{Methodology}
\label{chapter2}

\section{Methodology overview}
\label{sec:methodology-overview}

The methodology combines a controlled apparatus that varies incentive alignment, agent count, and reward formulation within a single simulator, a measurement substrate that isolates incentive structure from algorithmic variance, and a phased experimental sequence with promotion gates so claims at later phases rest on verified foundations.

\section{Simulator design}
\label{sec:simulator-design}

The simulator models a race as a sequence of discrete decision points at overtaking zones, rather than modelling lap-time physics at the second-by-second level \citep{heilmeier2018racesim}. This quasi-static approach keeps each race cheap enough to support multi-seed sweeps across $\alpha$, game structure, and agent count.

Zone difficulties are grounded in real race data. Using the FastF1 library, per-driver position changes from the 2023 Belgian Grand Prix were plotted against track distance. Position changes cluster at specific points on the circuit, indicating where overtaking happens in practice (Appendix~\ref{fig:overtakes-spa}). Because position changes are recorded at timing loops rather than exact on-track position, these values capture relative difficulty across zones rather than precise overtake points.

The analysis gives nine zones at Spa-Francorchamps. La Source (the hairpin at the start of the lap) has a difficulty of 0.2, reflecting how frequently overtakes happen there. The remaining zones sit between 0.7 and 0.9, consistent with how rare overtakes are at those locations in real racing. The full mapping from track location to zone is in Appendix~\ref{fig:overtaking-mapping}.

The simulator supports three stochasticity levels (s0, s1, s2) for robustness testing, since RL evaluation is sensitive to environmental variance \citep{henderson2018deep, agarwal2021statistical, sankary2018stochastic}. The levels differ only in the standard deviation $\sigma$ of a Gaussian noise term $\epsilon$ applied to the overtake success probability. s0 uses $\sigma = 0$ as the primary comparison track. s1 uses $\sigma = 0.02$, introducing modest outcome variance while preserving zone difficulty ordering. s2 uses $\sigma = 0.05$, which is sufficient to flip borderline outcomes while still preserving zone ordering in expectation.

\subsection{Overtake success probability}
\label{sec:overtake-prob}

When an agent chooses to attempt an overtake, the simulator resolves the outcome by computing a success probability and sampling from it. The probability is computed as

\[
P_{\text{success}} = \text{clip}\!\left(\, p_{\text{base}} + \delta_{\text{risk}} + \delta_{\text{gap}} + \epsilon,\ p_{\min},\ p_{\max} \right)
\]

where each term is defined as follows.

\textbf{Base probability.} The zone difficulty $d \in [0,1]$ controls the base success rate. An easy zone with $d = 0.2$ yields $p_{\text{base}} = 0.8$, while a hard zone with $d = 0.9$ yields $p_{\text{base}} = 0.1$.
\[
p_{\text{base}} = 1 - d
\]

\textbf{Risk adjustment.} The agent selects from three risk levels, each adding a fixed offset to the probability.
\[
\delta_{\text{risk}} = \begin{cases}
-0.08 & \text{CONSERVATIVE} \\
\phantom{+}0.00 & \text{NORMAL} \\
+0.12 & \text{AGGRESSIVE}
\end{cases}
\]
A more aggressive attempt therefore shifts the probability upward but also incurs a larger failure penalty in the reward signal, creating a risk-reward trade-off for the agent to learn.

\textbf{Gap modifier.} A closer following distance increases the probability of a successful overtake the closer you are to the car in front. The gap $g$ in kilometres is normalised against a reference distance of 0.1\,km, giving
\[
\delta_{\text{gap}} = \left(1 - \frac{g}{0.1} - 0.5\right) \times 0.3
\]

\textbf{Stochastic noise.} Gaussian noise $\epsilon \sim \mathcal{N}(0, \sigma)$ is added to the probability before clipping, with $\sigma$ set by the active stochasticity level. At s0 ($\sigma = 0$) the success probability is fixed for a given state, though outcomes still resolve through Bernoulli sampling.

The final probability is clipped to $[0.02,\ 0.95]$ to avoid degenerate certainty in either direction.

A conservative attempt reduces the success probability by 0.08 relative to normal, while an aggressive attempt raises it by 0.12. However, failure penalties in the reward signal scale in the opposite direction. A failed conservative attempt costs $-0.5$, a normal failure $-1.0$, and an aggressive failure $-1.5$. The optimal risk level therefore depends on zone difficulty, current gap, and positional context.

\subsection{Finish-first reward contract}

The reward signal combines four components.

\[
R = R_{\text{outcome}} + R_{\text{persistent\_position}} + R_{\text{tactical}} + R_{\text{penalty}}
\]

The objective hierarchy prioritises the finishing position. Outcome reward is terminal and dominant (weight 2.0), rewarding start-to-finish position gain. Persistent-position reward provides dense shaping from position changes that survive decision resolution (weight 0.1). Tactical reward gives local feedback for overtake outcomes at the risk-scaled failure costs listed above (weight 0.05). Penalty reward is a defensive component for unfinished races at weight 2.0 and rarely fires in practice because races typically complete.

Each experiment runs over five laps at Spa-Francorchamps. Agent composition varies by phase, ranging from one DQN plus the Base Agent up to four DQN agents plus Base, with the exact configuration per phase given in Table~\ref{tab:phases}.

The Base Agent is a fixed stochastic heuristic comparator. It only considers overtaking when a car ahead is within 100\,m and an overtaking zone is imminent. Within that window, it samples a risk level (conservative, normal, or aggressive) using zone difficulty and gap information, then samples whether to attempt the pass from a risk-dependent and difficulty-dependent probability. The agent does not update over time, which gives DQN policies a stationary non-learning baseline to be evaluated against.

\subsection{Design parameter choices}
\label{sec:design-parameters}

Several numerical values in the simulator specification (risk adjustments, failure penalties, reward weights, clipping bounds, stochasticity sigmas, gap modifier scaling, episode budgets, lap counts) are calibration choices made during early development. What is load-bearing for the methodology is the structural relationships between them, not the magnitudes. The risk-adjustment ordering, the failure-penalty inverse ordering, the reward weight hierarchy, and the s0 / s1 / s2 variance progression must all hold for the methodology to work. Specific values could differ without changing the conclusions. The structural design choices themselves (the zero-sum to non-zero-sum game structure switch, the agent-count progression, the $\alpha$ sweep across $[0,1]$, the curriculum warm-up then ramp then hold, and the Phase~10 mean-field approximation) are described in their respective phase subsections.

\section{Experimental methodology}
\label{sec:marl-methodology}

\subsection{Phase-to-RQ mapping}
\label{sec:phase-rq-mapping}

Each phase in the experimental programme either answers a research question from Chapter~\ref{chapter1}, or establishes a prerequisite for phases that do. The rightmost column of Table~\ref{tab:phases} shows the mapping.

Phase~0 and Phase~1 are validation gates that confirm the simulator is correct and a learning signal exists before any experimental claim is made. Phase~2 selects the DQN substrate that all subsequent phases use, so that algorithm differences do not confound incentive-structure comparisons. Phase~3 onwards carry the research-question load.

Several phases use an ablation design to isolate which specific mechanism drives a given effect. An ablation holds everything constant across conditions except the mechanism under test, so that any outcome difference is attributable to that mechanism. Phases~3, 5, and 7B all use this pattern for their respective isolations, detailed in each phase's subsection.

Phase~9 sits outside the RQ structure because its question is not about cooperation dynamics but about whether the learning problem genuinely requires adaptive strategy. If a zero-shot Large-Language Model (LLM) matches a trained DQN agent, the basis for the MARL investigation is weakened, so Phase~9 is a validity check on the project as a whole.

\subsection{Phase validation and promotion criteria}
\label{sec:promotion-gates}

Each phase is validated against criteria appropriate to its scope before downstream phases proceed. Early phases carry heavier gates because they establish the foundations for everything that follows.

Three formal gates govern the early phases. Gate~G0 is a hard automated integrity gate where all 12 simulator, reward, and telemetry tests pass or the gate fails (Appendix~\ref{appendix:phase01}). Gate~G1 is a methodological smoke gate verifying that a positive learning signal exists before the experimental matrix is committed to. Gate~G2 is a formal benchmark promotion gate that selects the DQN variant used for Phases~3 onwards.

Phase~3 and later phases carry phase-specific validation rather than uniform gates. Each phase has documented pass criteria tied to its experimental role, with the pass decision recorded alongside the phase analysis. A failed validation does not proceed. The phase is corrected and re-run under the same protocol. Validation is part pre-committed (pass criteria defined before the phase runs) and part investigation-driven (additional robustness checks or bias controls added in response to what the phase data shows). Both are reported alongside the phase results in Chapter~\ref{chapter4}.

\subsection{Measurement plan}
\label{sec:measurement-plan}

Each phase is evaluated at three stochasticity levels. s0 is the primary comparison track with noise removed. s1 and s2 are robustness tracks testing whether the s0 ranking survives under increasing outcome uncertainty. Within each level, multiple seeds estimate the variance of the learning process. A seed fixes the pseudo-random streams driving network initialisation, exploration, and replay sampling, so two runs at the same seed produce identical results. Different seeds produce different learning trajectories because RL is path-dependent, and multiple seeds therefore give a distribution of outcomes that is the basis for 95\% confidence intervals (CI95) across trials \citep{henderson2018deep}. Three seeds per evaluation cell is the design baseline, which is on the low end of current RL practice and widens confidence intervals relative to five- or ten-seed designs.

The core performance metrics are win rate, joint beat-base rate, per-agent beat-base rate, and positional advantage. Win rate is the proportion of evaluation races won by the relevant DQN agent, with 95\% CIs over evaluation races. Joint beat-base rate is the proportion of races in which every active DQN agent finishes ahead of the Base adversary, and becomes the central cooperative advantage metric from Phase~5 onwards. Per-agent beat-base rate applies the same test individually, distinguishing joint improvement from one agent carrying the result. Positional advantage is the mean finishing-position gap between two DQN agents.

Strategy is characterised through zone differentiation and risk differentiation indices. Zone differentiation is the mean absolute difference in per-zone attempt rates between the two agents. Risk differentiation is the mean absolute difference in attempt-conditioned risk proportions (the fraction of attempts at each risk level, computed over attempted overtakes only). These test RQ1's prediction of asymmetric role development.

\subsection{Phase 0: System integrity validation}
\label{sec:phase0-design}

Phase~0 validates the simulator, reward accounting, stochasticity implementation, and telemetry contract before any learning claims are made. Test Group~A checks simulator correctness, including position tracking, lap counting, and zone sequencing. Group~B validates stochastic probability distributions using repeated sampling to confirm the noise term produces the expected spread at each level. Group~C verifies the telemetry schema and reward component accounting.

\subsection{Phase 1: Single-agent smoke validation}
\label{sec:phase1-design}

Phase~1 runs vanilla DQN under the smoke protocol (200 training episodes, three seed sets of 20 evaluation races per seed) to confirm that a positive learning signal exists before the full Phase~2 budget is committed. Three seeds are used at s0 and one seed each at s1 and s2. Gate criteria require win rates above 0.5 across all s0 seeds with the confidence interval lower bound also clearing 0.5, zero fairness violations, and interpretable degradation across stochasticity levels.

\subsection{Phase 2: Single-agent DQN-family baseline}
\label{sec:phase2-methodology}
Phase~2 establishes the DQN-family reliability baseline under a controlled single-agent contract. Each variant (vanilla, double, dueling, rainbow-lite) is trained and evaluated against the fixed Base Agent with identical reward schema, observation contract, stochasticity level, episode budgets, and seeds \citep{henderson2018deep}. Phase~2 produces the per-algorithm stochasticity profiles that inform substrate selection for all subsequent phases. The shared training configuration was determined by systematic hyperparameter tuning prior to Phase~2, so no variant is advantaged by an arbitrary configuration choice.

\subsection{Phase 3: Independent learner MARL}
\label{sec:phase3-methodology}

Phase~3 replaces the fixed Base Agent with a second concurrent DQN learner. Both agents train at the same time, each with its own replay buffer and network parameters. This is the independent Q-learning (IQL) formulation, the standard value-based MARL baseline where each agent treats other agents as part of a non-stationary environment \citep{foerster2017stabilising, hernandezleal2019survey}. Phase~3 is gated on Phase~2 passing all promotion criteria.

\subsection{Phase 4: Alpha ($\alpha$) reward mixing}
\label{sec:alpha-mixing}

Phase~4 groups the two DQN agents into a single team and introduces a reward-sharing mechanism controlled by the coefficient $\alpha$. Each agent's effective reward is a weighted combination of its own reward and its teammate's reward.

\[
R_{\text{mixed}} = (1 - \alpha) \cdot R_{\text{own}} + \alpha \cdot R_{\text{teammate}}
\]

At $\alpha = 0$ each agent receives only its own reward, reproducing the Phase~3 IQL setup. At $\alpha = 1$ each agent receives only its teammate's reward. Intermediate values produce partial alignment. The team structure carries through every subsequent phase.

Five $\alpha$ values are tested in Phase~4 (0.0, 0.25, 0.50, 0.75, and 1.0) to identify whether there is a threshold beyond which cooperative incentive causes learning to collapse rather than cooperate. This sweep distinguishes a gradual performance degradation at high $\alpha$ from a structural passivity failure, which is the core empirical question of RQ2.

\subsection{Phase 5: Non-zero-sum game design}
\label{sec:nonzerosum}

Phases~2 to~4 use a two-agent zero-sum race where one agent's gain is always the other's loss. Phase~5 breaks this constraint by adding a fixed Base adversary as a third competitor, so both DQN agents can improve simultaneously by outperforming the adversary. An ablation within Phase~5 replaces each DQN agent with a Base agent in turn, isolating whether game structure or policy quality drives any observed improvement.

\subsection{Phases 6 and 7: Multi-team scaling}
\label{sec:multiteam}

Phase~6 extends the non-zero-sum setting to four DQN agents grouped as two teams of two, plus a Base adversary (five total agents on track). This tests whether the cooperative advantage from Phase~5 survives scaling.

Phase~7 splits into two sub-phases. Phase~7A tests three DQN agents plus the Base adversary (four total) to identify the agent count at which cooperation transitions from beneficial to harmful. Phase~7B revisits the five-agent setting with a low-$\alpha$ sweep to test whether any $\alpha$ value can recover cooperative performance at that scale. Together, Phases~6 and~7 test whether Phases~4 and~5's findings generalise beyond two agents.

\subsection{Phase 8: Curriculum Alpha ($\alpha$) scheduling}
\label{sec:curriculum}

Phase~8 applies a three-stage training schedule rather than a fixed $\alpha$ throughout. Agents train for 100 episodes at $\alpha = 0$ to establish individual policies, $\alpha$ increases linearly over 300 episodes to the target, and a final 100 episodes hold at the target. This tests whether the passivity failure at fixed $\alpha = 0.75$ reflects cooperative reward overwriting individual policies before they stabilise, as opposed to a structural incompatibility between cooperative incentives and independent learning. Phase~8 splits into Phase~8A (three DQN + Base) and Phase~8B (four DQN + Base) to test the curriculum at both scaling boundaries identified in Phases~6 and~7. The load-bearing element is the warm-up, ramp, and consolidation structure rather than the specific episode counts (Section~\ref{sec:design-parameters}).

\subsection{Phase 9: LLM semantic baseline}
\label{sec:llm-baseline}

Phase~9 introduces a zero-shot large language model as a non-learning comparator. The LLM is integrated into the simulator and queried automatically once per lap. Its prompt is programmatically constructed from simulator state and track configuration, but uses a fixed hand-authored system template that encodes domain knowledge, approximate probability heuristics, and strategic guidance. The model outputs a structured zone plan, which the simulator then executes using live gap information. No training, replay buffer, or gradient update is used.

The purpose is to establish whether domain reasoning with structured priors can match a trained DQN agent within this simulator's scope. Parity would indicate the learning problem is narrow enough to be solved by prompt-encoded reasoning, qualifying how broadly the MARL findings can be read, while a meaningful gap would indicate that learned policies capture strategic structure prompt-encoded reasoning cannot.

\subsection{Phase 10: Difference rewards}
\label{sec:difference-rewards}

Phase~10 tests difference rewards as an alternative cooperative formulation at the three-DQN-plus-Base configuration (four total agents). $\alpha$ mixing is a heuristic that does not formally separate individual contributions, so its passivity failure is fundamentally a credit assignment problem. Difference rewards address this by giving each agent credit only for its counterfactual contribution to collective performance \citep{wolpert2001optimal}. Computing the true counterfactual is costly, so a mean-field approximation is used where each agent's reward is its position delta plus the deviation from the team mean.

\[
R_{\text{diff},i} = \Delta_i + (\Delta_i - \bar{\Delta}_{\text{team}})
\]

Whether this approximation preserves the cooperative gradient property is the central question of Phase~10.

\section{Project management}

\subsection{Sprint structure and deliverable mapping}

Each sprint covers one phase and produces three artefacts. A protocol documents the phase design and gate criteria, an analysis captures results and interpretation, and a dataset artefact contains the evaluation outputs.

The five project deliverables map to sprint groups. Deliverable~3 (minimal viable product with basic metrics) corresponds to Sprints~0 and~1. Deliverable~4 (expanded state space, varied parameters, and algorithm comparison) corresponds to Sprints~2 through~9. Deliverable~5 (final report with comparative analysis) is the present document. The gate structure means each deliverable is verifiable at its own checkpoint rather than assessed only at final submission.

\subsection{Version control and gate discipline}

Version control uses Git with one feature branch per phase. A phase branch starts from the current main, carries all development for that phase, and merges back only after the phase's gate or validation criteria are met. This rule means the main branch always reflects a gate-passed simulator state, and any commit on main can be checked out and reproduced in full.

Commit messages document phase-level findings at meaningful checkpoints. These include zone behaviour observations, collapse events, and implementation details that affected results. The commit history therefore doubles as an engineering log, with each phase's development trail preserved for later audit.

\subsection{Reproducibility through telemetry}

Every run embeds its executed configuration in the output metrics. The active configuration, training and evaluation budgets, and seed assignments are serialised into the recorded output, so any experiment can be reproduced from that record without recovering the repository state at the time. Downstream analysis reads the algorithm, hyperparameter, and scenario fields from the telemetry output rather than from the config file on disk, so any mismatch between what was meant to run and what actually ran shows up during analysis.

\subsection{Research question independence}

The phased structure is not tied to any one research question. The simulator, gate machinery, and evaluation infrastructure are built around stochasticity level, agent count, and reward formulation without picking one as primary. New phases extend the existing ones rather than replace them, and completed phases remain valid whatever direction the experimental matrix takes.
