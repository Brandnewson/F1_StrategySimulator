The simulator is implemented as a set of well-separated Python modules, each with a single responsibility. The entry point for a single train-then-eval trial resolves the active complexity profile from configuration and passes it to the simulator engine. A companion orchestration script sequences calls across the full algorithm comparison matrix.

The simulator engine owns the tick-by-tick race loop, maintaining race state that tracks all driver positions, lap counts, and time gaps at each simulation step. At each overtaking zone decision point, the engine constructs the observation vector for the approaching driver and queries the driver's agent for an action. The overtake resolution outcome is sampled from the probability function described in Section~\ref{sec:overtake-prob} and logged to telemetry. At the end of each episode the engine computes terminal reward components and triggers epsilon decay for each DQN-type driver.

All agents implement a common interface providing an action-selection method that the simulator calls uniformly. This abstraction means the simulator does not need to distinguish between algorithm variants at the engine level. It receives a structured action containing a risk level and an overtake attempt flag regardless of which variant is active.

The full UML class diagram for the simulator core is in Appendix~\ref{appendix:classdiagram}.

A real-time visualisation mode renders the circuit with agent positions, lap count, and time gap annotations, used during development to verify zone sequencing and reward log consistency. It is disabled for benchmark runs.

\subsection{Config-driven design as a fairness control}

All parameters that affect the experimental outcome are externalised to a single configuration file. This covers hyperparameters, the reward component schema and weights, the observation feature list, stochasticity presets, evaluation protocol budgets, and seed sets. No algorithm implementation reads or modifies these values directly. They are injected at construction time.

This design ensures that the only difference between a vanilla and a double DQN run is the learning algorithm itself. An external reviewer can fully reconstruct any comparison from the recorded output of each run, which embeds the executed configuration, budgets, and seed assignments.

\section{Agent implementations}

\subsection{DQN-family agents}

All four DQN variants are parameterisations of a single agent class, selected via a config field. This approach guarantees that all shared infrastructure is identical across variants. The training loop, action space, epsilon schedule, and feedback interface are all shared. Only the variant-specific elements differ. The shared training loop structure is illustrated in Appendix~\ref{appendix:dqnloop}.

\textbf{Vanilla DQN.} The vanilla variant uses a two-hidden-layer multi-layer perceptron (MLP) with ReLU (rectified linear unit, a standard non-linear activation function) activations and a linear Q-value output head. Q-values represent the expected future reward for taking an action in a given state. The hidden layer width is 512 units. Experience from past decisions is stored in a uniform circular replay buffer of capacity 150{,}000 and sampled randomly to update the network, which breaks correlation in training data. The target network is updated every 1{,}000 gradient steps by hard copy. The Bellman target (the value the Q-network is trained to predict) is computed as:
\[
y = r + \gamma \max_{a'} Q_{\text{target}}(s', a')
\]
where $r$ is the reward received after the action, $\gamma$ is the discount factor controlling how much future rewards are weighted relative to immediate ones, $s'$ is the next state, and $a'$ is the action taken in the next state. This is the baseline against which all other variants are compared \citep{mnih2015human}.

\textbf{Double DQN.} Vanilla DQN has a known problem where it tends to be over-optimistic about how good certain actions are, especially when rewards are noisy. Double DQN fixes this by splitting the decision into two steps. Instead of letting a single network both pick the best next action and estimate its value (which lets any over-estimation feed back into the target), the online network picks the action and the older target network grades it. This decouples action selection from evaluation, so random noise in the network's estimates stops compounding into systematic overestimation \citep{vanhasselt2016double}.
\[
y = r + \gamma Q_{\text{target}}\!\left(s',\, \mathop{\arg\max}_{a'} Q_{\text{online}}(s', a')\right)
\]
The architecture and uniform replay are otherwise the same as vanilla.

\textbf{Dueling DQN.} Dueling DQN makes learning more efficient by recognising that action choice does not always matter. In many states (for example, when the car ahead is far away), every action leads to roughly the same outcome, so the agent only needs to learn that the state is fine. The dueling architecture splits the network into two output heads. A state-value head estimates how good a state is overall, independent of action. An action-advantage head estimates how much better or worse each specific action is relative to the average. The Q-value is reconstructed as:
\[
Q(s,a) = V(s) + \left(A(s,a) - \frac{1}{|\mathcal{A}|}\sum_{a'} A(s,a')\right)
\]
Subtracting the mean advantage ensures identifiability of $V$ and $A$. This architecture improves learning efficiency in states where the choice of action has little effect on the outcome, which is common at overtaking zones where the car ahead is far away and \textsc{HOLD} is clearly dominant \citep{wang2016dueling}.

\textbf{Rainbow-lite.} Rainbow-lite combines the three previous variants with two additions, which together make it the strongest in the DQN family. It uses double DQN target computation, the dueling architecture, prioritised experience replay (PER), and multi-step returns. PER replays transitions that caused large prediction errors more often than transitions the agent already understands well, focusing learning effort where it is most useful \citep{schaul2016prioritized}. Multi-step returns let each update use rewards from the next three steps rather than just one, which propagates useful information through the network faster \citep{hessel2018rainbow}.
\[
y = r_t + \gamma r_{t+1} + \gamma^2 r_{t+2} + \gamma^3 \max_{a'} Q_{\text{target}}(s_{t+3}, a')
\]
PER samples transitions with probability proportional to $|TD_{\text{error}}|^\alpha$ where $\alpha = 0.6$, with importance-sampling weights annealed from $\beta = 0.4$ to $1.0$ over 100{,}000 steps to correct the introduced bias.

\subsection{Base agent}

The Base Agent is a gap-aware heuristic policy. At each decision point it identifies the upcoming overtaking zone, reads the zone difficulty and the gap to the car ahead, and builds a soft preference distribution over the three risk levels. Easier zones and closer gaps increase the weight on \textsc{AGGRESSIVE}. Harder zones and larger gaps increase the weight on \textsc{CONSERVATIVE}. An overtake attempt is then made with a probability that depends on the sampled risk level and the gap.

The Base Agent serves as a stable, reproducible fixed opponent for Phase 2. It does not learn and its policy does not change between episodes. This is a deliberate experimental control. If both competitors were learning simultaneously in Phase 2, improvements in the DQN agent's win rate could partly reflect opponent degradation rather than genuine policy improvement. A fixed opponent eliminates this confound and provides the clean single-agent baseline required before Phase 3 introduces concurrent learning.

\subsection{Shared training infrastructure}

The epsilon-greedy exploration schedule is shared across all variants. Epsilon-greedy is a simple exploration strategy where the agent selects a random action with probability $\epsilon$ and the best-known action otherwise. Epsilon starts at 1.0 (fully random) and decays multiplicatively each episode by a factor of 0.995 toward a minimum. With 500 training episodes (the Phase 2 budget), epsilon reaches approximately 0.08, meaning the agent selects a random action roughly 8\% of the time at the end of training. This represents a near-converged policy.

The \textsc{Adam} optimiser is used with a learning rate of $7 \times 10^{-4}$ across all variants. Adam is a widely-used gradient-based optimiser that adapts the step size for each network parameter based on the history of its gradients, which stabilises training when gradients vary in magnitude \citep{kingma2015adam}. Target network synchronisation occurs every 1{,}000 gradient steps for vanilla, double, and dueling. Rainbow-lite uses the same frequency for the hard target copy. Batch size is 64 throughout.

The shared hyperparameters were determined using Autoresearch \citep{karpathy2024autoresearch}, a hyperparameter tuning tool that iteratively proposes and evaluates single-parameter changes against a scored objective. Thirty iterations were run, each scored across three seeds with fifty training runs per seed, before the configuration was committed. Tuning was performed against the Phase~2 single-agent contract, so the hyperparameters reflect single-agent stability rather than being tuned for multi-agent dynamics. This keeps the comparisons between DQN variants fair within Phase~2 and avoids tuning bias that could favour any particular variant in later phases.

\section{Benchmark infrastructure}

\subsection{Single-trial runner}

The single-trial runner executes one train-then-eval trial for a given algorithm variant. It accepts arguments for the training budget, evaluation budget, training seed, evaluation seeds, complexity profile, stochasticity level, and output path. The script first runs training episodes with the agent in exploration mode, then freezes the policy and runs evaluation episodes with exploration disabled. At the end of the run it writes a structured metrics file containing the win rate, objective score, fairness diagnostics, and full behavioural diagnostics including the per-zone attempt and success breakdown and risk distribution.

\subsection{Benchmark matrix orchestrator}

The benchmark orchestrator sequences single-trial runs across the full comparison matrix. The matrix contains every combination of algorithm, seed, and stochasticity level. Each combination runs as an independent trial.

The orchestrator checks fairness against a declared contract. The declared contract specifies the training and evaluation budgets, seed sets, complexity profile, and stochasticity level that all algorithms must share. If any violation is detected, the orchestrator aborts. On completion it writes per-algorithm statistics with 95\% confidence intervals and a summary table for rapid inspection.

A companion aggregation script collects the per-seed metrics files produced by Phase~2 and computes means, standard deviations, and 95\% confidence intervals across seeds. The Phase~2 algorithm comparison in Chapter~\ref{chapter4} is derived from this script. Later phases use the MARL evaluation runner described in the next subsection.

\subsection{Telemetry and logging}

Every race produces a JSON Lines (JSONL) entry containing the full per-driver result. Each entry includes the driver name, final race position, total reward, per-component reward totals, stochasticity level, complexity profile, and a \texttt{decision\_events} array. Each decision event records the zone identifier, lap number, action label, attempt flag, success flag, success probability, reward received, individual reward component values, and a \texttt{reward\_component\_sum\_error} field that verifies the arithmetic consistency of the reward decomposition. The telemetry schema is validated in Phase 0 Group~C tests and is described in detail in Appendix~\ref{appendix:phase01}.

\subsection{MARL evaluation runner}

A separate evaluation runner handles MARL trials from Phase~3 onwards. Each trial loads two trained DQN policies into fixed evaluation slots labelled A1 and A2, where the A1 slot is the first-defined position and A2 is the second. Agent~1 refers to the trained policy in the A1 slot, and Agent~2 to the trained policy in the A2 slot. The primary metric is the fraction of evaluation races in which Agent~1 finishes ahead of Agent~2. The runner also computes a zone differentiation index, the mean absolute difference in per-zone attempt rates between the two agents, and a risk differentiation index, the mean absolute difference in attempt-conditioned risk proportions. A non-stationarity drift metric records the change in the Agent~1 win rate between the first and last third of the evaluation window, providing evidence of whether the advantage relationship is stable across the evaluation window. From Phase~5 onwards the runner was extended to record team-level metrics, including the joint beat-base rate, the fraction of races in which all active DQN agents finish ahead of the Base adversary, which measures cooperative performance rather than relative DQN performance.

A crossplay check was run for rainbow-lite at s2 across three seeds to validate the evaluation protocol. Crossplay evaluation tests trained policies against each other in controlled configurations to detect confounds in the evaluation setup. Each MARL trial loads two trained agents into fixed evaluation slots, labelled A1 and A2, where A1 is the first-defined slot and A2 is the second. In the self-play condition both slots are loaded with the same trained weights, so any systematic deviation from a 0.500 A1 win rate indicates positional bias. Positional bias is any systematic advantage arising from which slot an agent is assigned rather than from policy quality. In the swapped condition the A1 and A2 model files are exchanged, isolating policy quality from slot assignment. The crossplay results are presented in Section~\ref{sec:phase3-bias} and incorporated into the validity threat discussion in Section~\ref{sec:validity}.

\subsection{MARL and phase extensions}
\label{sec:phase-extensions}

Phase~3 required the codebase to support two concurrent DQN learners. The complexity profile layer was extended with a two-DQN competitor selector that initialises both agents from the same hyperparameter block, each with separate replay buffers, network parameters, and epsilon schedules. The simulator engine required no changes, as it already triggers a training step on every DQN-type driver at the end of each episode.

Phases~4 to~10 required further extensions, all controlled through configuration with no changes to agent learning code. The reward calculation was extended with an $\alpha$ blending step; setting $\alpha$ to zero reproduces the original reward contract exactly. A team-grouping selector was added to assign agents to named teams for $\alpha$ mixing. A curriculum scheduler was implemented as a thin wrapper around the training loop, reading the target $\alpha$ and episode boundaries from config and hooking into the existing episode-end callback. A difference reward mode computes the mean-field approximation described in Section~\ref{sec:difference-rewards}. The LLM agent implements the same common interface as all other agents. Its prompt is programmatically constructed from simulator state and track configuration, using a fixed hand-authored system template that encodes domain knowledge, probability heuristics, and strategic guidance (Section~\ref{sec:llm-baseline}). The agent queries Claude Haiku, a language model API provided by Anthropic, once per lap and outputs a structured zone plan. On parse failure it retains the previous lap's plan or installs a default plan, which still allows attempts at easy zones.

\subsection{Challenges and resolutions}

Three significant challenges were encountered during implementation. First, the initial Base Agent implementation had a missing gap attribute that caused near-zero attempt probability. This surfaced during Phase~0 testing rather than being caught by face-value inspection, validating the value of automated integrity tests. Second, the telemetry reward arithmetic check revealed a reward-sum accounting error in the reward decomposition that would have remained hidden without the dedicated arithmetic verification field. Third, a Phase~10 configuration specified the wrong algorithm variant. The error was caught during analysis by reading the algorithm field from the telemetry output rather than the config file, confirming the value of config snapshots embedded in run metrics. Each case points to the same methodological pattern. Integrity machinery that reads from output rather than input catches silent-failure bugs that face-value checks miss.

All Phase~0 assertions are formalised as automated tests. The full test inventory and defect log are in Appendix~\ref{appendix:phase01}. All 12 tests pass across the three groups. Two defects were caught and corrected during Phase~0 development, with details in the appendix. Neither would have been detectable from win rates alone. The system passed all integrity checks and was cleared to proceed to Phase 1.

\section{Phase 1: Single-agent smoke validation}

Vanilla DQN is evaluated under the Phase~1 protocol (200 training episodes, three seed sets of 20 evaluation races per seed) across three s0 seeds and one seed each at s1 and s2. All three s0 seeds produce win rates above the 0.50 gate threshold with zero DNF races and zero fairness violations. The confidence interval lower bound across seeds also clears 0.50, satisfying the stability criterion. Full per-zone and per-stochasticity breakdowns are in Appendix~\ref{appendix:phase01}.

Win rate is flat across s0 and s1. This masks an underlying shift in risk distribution between levels, and the flatness is partly explained by La Source dominating outcomes regardless of risk choice (Appendix~\ref{appendix:phase01}). The risk distribution shift is consistent with Q-value overestimation under noisy rewards \citep{vanhasselt2016double}, which motivates the Phase~2 double DQN comparison. Gate~G1 was cleared.
